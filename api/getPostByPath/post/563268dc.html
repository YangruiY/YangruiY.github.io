桥接模式：

\n
    \n
  • 可以将一个大类或者一系列紧密相关的类拆分为抽象的和实现两个独立的层次结构，在开发的时候进行分别使用
    • \n
  • 使用组合代替继承的方式进行解决问题，将原来的某个维度抽象成独立的类层次
    • \n
  • 可以在初始类中引用这个新层次的对象了，从而使类不必拥有所有的状态和行为
    • \n
\n

组成：

\n
    \n
  • 抽象部分层次—(GUI)：部分实体的高阶控制层，不是完成任何具体的工作
    • \n
  • 实现部分层次—(API)：具体的实现类，即平台
    • \n
\n
\n

所以可以在上面两个独立的方向进行扩展和应用；从而开发实现多个不同的GUI 和 API

\n
\n

结果：

\n
    \n
  • 无需改动与 API 相关的类就可以修改 GUI 类。 此外如果想支持一个新的操作系统， 只需在实现部分层次中创建一个子类即可
    • \n
\n

UML构成：

\n
    \n
  1. 抽象部分：定义抽象类控制高层逻辑
    2. \n
  2. 实现部分：所有具体的实现类声明通用的接口
    3. \n
  3. 具体的实现
    4. \n
  4. 进行精准的抽象： 控制逻辑的变体
    5. \n
\n
\n

个人理解：

\n

就是在 实现 和 抽象 之间进行 构建桥梁，这个桥梁就是 对其他维度的信息的引用；主要的目的是减少每个对状态和行为的拥有。

\n

针对每次都要进行对原有维度进行进一步的划分的时候可以使用这个模式

\n

比如 原来是在形状的基础上进行拆分，现在再加上颜色的维度进行划分

\n
\n

使用场景：

\n
    \n
  • \n

    拆分或者重组一个具有多重功能的类

    \n
    • \n
  • \n

    在几个独立的维度上进行扩展

    \n
    • \n
  • \n

    在运行时切换不用的实现方法

    \n
    • \n
\n

优点：

\n
    \n
  • 创建和平台无关的类和程序
    • \n
  • 仅和高层抽象部分进行互动，不接触平台的详细信息
    • \n
  • 符合开闭原则
    • \n
  • 符合单一职责原则
    • \n
\n

缺点：

\n
    \n
  • 不适合内聚度很高的代码
    • \n
\n
